# T-GEO 全栈开发交接说明书

> 面向：接手继续开发的你 + Claude Code
> 目标：把现在这套「漂亮但是假数据」的前端 demo，做成「真能跑诊断、出报告、管内容」的全栈产品。
> 配套文档：本仓库 `GEO_PRD_本地生活版.md`（产品全貌）、`HERO_VIDEO_PROMPT.md`（背景视频）、`CLAUDE.md`（Claude Code 上下文）。

---

## 0. 一句话现状

**现在 = 一个纯前端 SPA，所有数据都是写死的假数据（mock）。** 登录是假的、诊断是定时器演示、图表数据是常量、点击按钮弹的是前端提示。
**全栈开发 = 把这些假数据逐个替换成「真后端 + 真数据库 + 真大模型调用」。** UI 层基本不用重做，主要是「接管线」。

## 2. 目标全栈架构

### 2.1 推荐方案（与 PRD 的 vibe-coding 思路一致，改动最小）

**保留现有 Vite + React 前端，后端用 Supabase 兜底 + 一个独立 Worker 跑长任务。**

```
┌─────────────────────────────────────────────┐
│  前端 (本仓库, Vite SPA)                       │
│  React 19 + TanStack Query 取数 + Supabase JS │
└───────────────┬─────────────────────────────┘
                │ HTTPS / Supabase Client
┌───────────────┴─────────────────────────────┐
│  Supabase (BaaS，一站式)                       │
│  Postgres + Auth + Storage + pgvector + RLS   │
│  Edge Functions（轻量 API：CRUD、触发任务）     │
└───────────────┬─────────────────────────────┘
                │ 入队
┌───────────────┴─────────────────────────────┐
│  Worker 服务 (Railway / Fly.io 上的 Node)      │
│  任务队列(Inngest/Trigger.dev) → 跑诊断:        │
│  并发调用大模型 API + Playwright 爬 C 端平台     │
│  → 解析 → 算 T-LBR → 写回 Postgres → 通知前端    │
└───────────────┬─────────────────────────────┘
                │
┌───────────────┴─────────────────────────────┐
│  全渠道: DeepSeek/豆包/Kimi/元宝/通义千问/文心/  │
│  智谱/混元/讯飞 + ChatGPT/Gemini/Claude/Perplexity│
│  + 入口型: 百度AI/夸克/抖音搜/小红书搜/微信搜      │
│  (有API直调; 无API的C端用 Playwright 爬)         │
└─────────────────────────────────────────────┘
```

**为什么这样选**：
- Supabase 一个服务把「数据库 + 登录 + 文件存储 + 向量检索 + 多租户隔离」全包了，最省事。
- 诊断要并发跑「100 意图 × 10 平台 × 3 次 ≈ 3000 次调用」，耗时长，**必须放后台 Worker + 队列**，不能放在前端或短时函数里。
- 前端几乎不用重写，只是把 mock 换成接口。

### 2.2 备选方案（暂不推荐现在做）

迁移到 **Next.js**（前后端同仓、API Routes、SSR）。优点是部署统一、SEO 友好；缺点是要把现在的 Vite SPA 重写一遍。**建议等到 SaaS 阶段、要做营销官网 + SEO 时再迁**，现在先用方案一把后端跑通。

## 3. 数据库设计（MVP 精简版，可直接建表）

> 以下是 PRD §11「25 张表」的 MVP 子集，够跑通「诊断→报告→知识库→内容→看板」闭环。建在 Supabase 的 Postgres 上，全部开启 RLS 多租户隔离。

```sql
-- 启用向量扩展（知识库 RAG 用）
create extension if not exists vector;

-- ① 租户与用户（多租户隔离的根）
create table tenants (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  plan text default 'self_op',          -- self_op自营 / saas
  created_at timestamptz default now()
);

create table users (
  id uuid primary key,                   -- = auth.users.id
  tenant_id uuid references tenants(id),
  email text,
  role text default 'owner'              -- owner/admin/editor/viewer
);

-- ② 客户与店铺
create table shops (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants(id),
  name text not null,
  industry text not null,                -- restaurant / spa / hotel
  city text, district text,
  brand_info jsonb,                      -- 店铺基础信息
  created_at timestamptz default now()
);

-- ③ 行业模板（差异化的核心）
create table industry_templates (
  id uuid primary key default gen_random_uuid(),
  industry text not null,
  version int default 1,
  intents jsonb,        -- 用 src/lib/intents.ts 的 buildIndustryTemplateSeed(industry) 播种（存模板+变量池，按店再 expandIntents 展开）
  schema jsonb,         -- 知识图谱字段结构
  prompts jsonb         -- 该行业默认 prompt 集
);

-- ④ 诊断
create table diagnoses (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  status text default 'queued',          -- queued/running/done/failed
  score_lbr numeric,
  breakdown jsonb,                       -- 4 维度子分
  started_at timestamptz, finished_at timestamptz
);

create table intent_queries (
  id uuid primary key default gen_random_uuid(),
  diagnosis_id uuid references diagnoses(id),
  intent_text text,
  platform text,                         -- deepseek/kimi/doubao...
  run_no int,
  response_text text,
  parsed jsonb                           -- 是否提及/排名/描述/信源/竞品
);

create table lbr_scores (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  score numeric,
  breakdown jsonb,
  date date default current_date
);

-- ⑤ Prompt 库
create table prompt_templates (
  id uuid primary key default gen_random_uuid(),
  tenant_id uuid references tenants(id),
  code text, industry text, scenario text,
  body text, variables jsonb,
  version int default 1, parent_id uuid,
  metrics jsonb
);

create table prompt_runs (
  id uuid primary key default gen_random_uuid(),
  template_id uuid references prompt_templates(id),
  shop_id uuid references shops(id),
  output text, quality_score numeric, tokens int, model text,
  created_at timestamptz default now()
);

-- ⑥ 知识库（RAG）
create table kb_documents (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  type text, source text, raw_content text,
  created_at timestamptz default now()
);

create table kb_facts (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  kg_path text,            -- 如 signature_dish.name
  value text,
  confidence numeric,
  source_doc_id uuid references kb_documents(id),
  embedding vector(1536)   -- 语义检索
);

-- ⑦ 内容生产
create table contents (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  type text,                             -- 小红书/问答/短视频脚本...
  prompt_template_id uuid,
  body text,
  compliance text default 'pending',     -- pass/needs_fix
  status text default 'draft'            -- draft/ready/distributed
);

-- ⑧ 分发
create table distribution_tasks (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  content_id uuid references contents(id),
  platform text, status text default 'backlog',  -- backlog/review/executing/published
  due_at timestamptz, published_url text
);

-- ⑨ 监测（看板数据来源）
create table mention_records (
  id uuid primary key default gen_random_uuid(),
  shop_id uuid references shops(id),
  platform text, intent_text text,
  mentioned bool, rank int, snippet text, source_cited text,
  captured_at timestamptz default now()
);
```

**RLS 多租户隔离示例**（每张含 tenant_id 的表都加）：

```sql
alter table shops enable row level security;
create policy tenant_isolation on shops for all
  using (tenant_id = (auth.jwt() ->> 'tenant_id')::uuid);
```

## 5. AI 诊断引擎（最硬核的部分）

这是产品的灵魂，独立成一个 Worker 服务。流程（对应 PRD §8.1 / §12.3）：

```
1. 前端 POST /diagnoses → 写一条 diagnoses(status=queued) → 入队
2. Worker 取任务:
   2.1 读 shop → 调 src/lib/intents.ts 的 expandIntents(industry, shopCtx) 展开意图
       （餐饮123/SPA104/酒旅106 条，已带 priority 与 topRec 标记；可再叠加 LLM 扩写长尾）
   2.2 ★从 src/lib/platforms.ts 读全渠道清单（18 个）★
       拆成 100 意图 × 渠道数 × 3 次 任务（≈ 5400 条，按渠道并发上限节流）
   2.3 按 platform.hasApi 路由到对应「渠道适配器」:
       - hasApi=true  → API 适配器直接调（DeepSeek/豆包/Kimi/通义千问/文心/智谱/混元/讯飞/ChatGPT/Gemini/Claude/Perplexity）
       - hasApi=false → 爬虫适配器 Playwright+代理池（元宝/百度AI/夸克/抖音搜/小红书搜/微信搜），接受 ~85% 成功率 + 重试
   2.4 每条结果 NLP 解析: 是否提及品牌 / 排名 / 描述准确度 / 引用信源 / 竞品 → 写 intent_queries(platform=渠道id)
   2.5 算 T-LBR 总分 + 4 维度 → 写 lbr_scores、更新 diagnoses
   2.6 (可选) 生成报告正文 → 存储
3. diagnoses(status=done)，Realtime 通知前端刷新
```

**渠道适配器模式（关键工程约定）**：诊断引擎**不要散落 if-else 判断平台**，而是为每个渠道实现统一接口，由 `platforms.ts` 驱动。这样新增一个 AI 平台 = 加一条 `platforms.ts` 记录 + 写一个适配器，**诊断主流程零改动**。

```ts
// worker/channels/types.ts —— 全渠道统一接口
export interface ChannelAdapter {
  id: string;                       // 对应 platforms.ts 的 id
  query(intent: string): Promise<{ answer: string; citations?: string[] }>;
}

// worker/runDiagnosis.ts —— 主流程按 platforms.ts 自动铺开，无需改动
import { AI_PLATFORMS } from '../src/lib/platforms';
import { getAdapter } from './channels';   // 按 id 取 API 或 爬虫适配器

const RUNS = 3;                      // 每个意图每渠道跑 3 次消除随机性
for (const intent of intents) {
  for (const p of AI_PLATFORMS) {    // ← 18 个渠道全自动覆盖
    const adapter = getAdapter(p);   // hasApi 决定返回 API 还是 Playwright 适配器
    for (let run = 1; run <= RUNS; run++) {
      enqueue(() => adapter.query(intent)
        .then(r => parseAndStore(diagnosisId, p.id, intent, run, r)));
    }
  }
}
```

> **MVP 起步建议**：先只实现 `hasApi=true` 的 4–5 个主力渠道（DeepSeek/通义千问/文心/Gemini/ChatGPT）跑通闭环，再逐个补爬虫渠道。`platforms.ts` 里 `hasApi=false` 的渠道在适配器没就绪前，主流程会自动跳过/标记 pending，不阻塞上线。

**T-LBR 评分公式**（PRD §8.1.2.3，直接实现）：

```
LBR = 0.35×提及率 + 0.25×排名得分 + 0.20×描述准确度
    + 0.10×信源权威度 + 0.10×情感倾向
排名得分：第1=10, 第2=8, 第3=6, 第4-5=4, 其后=1
展示 4 维：曝光 / 定位 / 信任 / 转化倾向
```

**成本与稳定性**（PRD §3.2）：调用量 = 意图数 × 渠道数 × 3。18 渠道全开约 5400 次/店，成本随渠道数线性增长——**MVP 先只开 `hasApi=true` 的 4–5 个主力渠道**控本（约 1500 次、几十元），跑顺再逐步放开。用 DeepSeek 跑量、Claude/通义 提质、GPT 兜底。爬虫渠道接受 80–90% 成功率 + 定时重试，不追求 100%。

**队列选型**：Inngest 或 Trigger.dev（托管、vibe-coding 友好）；自托管可用 BullMQ + Redis。

## 6. 前端如何从 mock 切到真实 API（落地手法）

1. **装数据层**：`npm i @tanstack/react-query @supabase/supabase-js`，在 `main.tsx` 包一层 `QueryClientProvider`。
2. **建 Supabase 客户端**：`src/lib/supabase.ts`，读 `import.meta.env.VITE_SUPABASE_URL / VITE_SUPABASE_ANON_KEY`。
3. **建数据 hooks**：如 `src/hooks/useLbr.ts`、`useDiagnose.ts`、`useContents.ts`，内部用 `useQuery/useMutation`。
4. **逐页替换**：把页面顶部的 `const data = [...]` 删掉，改成 `const { data } = useLbrTrend(shop.id)`，加载态用现成的科幻 loading 动画。
5. **登录**：`App.tsx` 的假表单换成 `supabase.auth.signInWithPassword()`；保留那段开机动画当过场。
6. **诊断**：`一键智能诊断` 改成 `POST /diagnoses` + 订阅状态，动画进度绑真实 `status`。

> 顺序建议：**Dashboard 先打通**（最能体现"真活了"），再 Reports、Knowledge、ContentFactory、Distribution、DataBoard、Settings。

## 6.5 运营投喂层（把「生产+监控」串成可执行作战排期）

这是 demo 之前缺的关键层：内容生产完，要回答「一天发多少篇 / 在哪发 / 谁发 / 需要多少账号 / 怎么发」。

**事实源 `src/lib/distribution.ts`**（可配置、会演进）：
- `FEED_PLATFORMS`：每个平台的 AI 引用权重、单账号每周安全发布上限、发布主体类型(客户自营/真实达人/运营)、限定行业。
- `STAGE_WEEKLY_TOTAL`：冷启动/爬坡/维护 三阶段的每周总篇数。
- `generateWeeklyPlan(industry, stage)`：按权重最大余数法把「本周 N 篇」分配到各平台，并按「篇数 ÷ 单账号安全频率」倒推账号矩阵。

**前端**：分发看板页(`Distribution.tsx`)顶部据此渲染「本周投喂作战卡」——总篇数 / 各平台分配(在哪发·发几篇·谁发) / 账号矩阵(需要几个号) / 完成进度；阶段可切换、节奏随之重算。任务卡补了发布主体徽章 + 发布后「回填 URL」入口。

**闭环（后端要接）**：诊断短板 → 计划引擎排期 → 任务执行 → 回填发布 URL → 进监测(是否被 AI 引用) → 效果反馈下一轮加权。落库：`distribution_tasks` 增加 `publisher / target_intent / due_at / published_url / cited` 字段。

**合规**：全是客户自营账号 + 真实达人(人机协作)，绝不养号/刷量。

## 6.6 Agent 内容流水线（内容工厂 ↔ 分发看板 打通）

**致命缺口已补**：内容工厂不再是独立的"内容仓库"，而是被「投喂作战计划」驱动的一条 Agent 流水线，与分发看板共享同一份数据。

**事实源 `src/lib/workflow.ts`**：
- `ContentJob`：一条内容的全生命周期（`draft → ready → backlog → review → executing → published`），从"计划要发"到"被 AI 引用"只有一份数据、一个 id。
- `PIPELINE_STAGES`：6 阶段及其 Agent —— 计划引擎 → 内容Agent(生成) → 合规Agent(校验) → 一键采用 → 派单分发 → 监测Agent(回填/被引)。
- `buildJobsFromPlan(industry, stage, shopName)`：把 `distribution.ts` 的作战计划 + `intents.ts` 的目标意图，编排成内容任务。

**共享状态 `src/store/PipelineContext.tsx`**（前端）：
- 内容工厂"按本周作战计划一键生成"→ 产出草稿 → 合规校验 → **「采用」→ 状态置 `backlog` → 立刻出现在分发看板**。
- 分发看板拖拽 = `moveTask` 改同一条数据；已发布卡「回填URL」= `backfill` → `cited`。
- 真正同步：两页是同一条流水线的两道工序，不是各自独立的 mock。

**后端要接**：`contents` 表用 `status` 承载生命周期；内容Agent = Worker 取计划项 → 调 LLM(知识库+Prompt) → 写草稿；合规Agent = 敏感词/广告法过滤；采用 = 写 `distribution_tasks`；回填 URL → 进 `mention_records` 监测闭环。

## 6.7 多 Agent 自我进化闭环（工作台 = 编排中枢，越用越聪明）

产品的灵魂：不是一堆功能页，而是一套会自我进化的多 Agent 工作流。

**事实源 `src/lib/agents.ts`**：
- `AGENTS` / `AGENT_LOOP`：7 个 Agent 及闭环 —— 诊断 → 计划 → 内容 → 合规 → 分发 → 监测 → **优化(回喂)** ↺ 计划/内容。
- `computeLearning(jobs)`：从已发布且**被 AI 引用**的内容里学出 —— 智能增益(0–100)、当前最优打法(平台×内容形态命中率)、尚未覆盖的短板意图。

**闭环怎么转（数据流）**：
1. **诊断报告** 找出未被推荐的短板意图 → 驱动 **分发计划**（distribution.ts）该补哪些、发哪、发几篇。
2. **内容工厂** 按"短板 + 已验证最优打法"生成内容（`buildJobsFromPlan` 接 `LearnedBias`）。
3. 发布后 **监测 Agent** 追踪是否被各 AI 引用 → 写回 `mention_records` / job.cited。
4. **优化 Agent**(`computeLearning`) 把"哪种打法真见效"沉淀为权重 → 回喂下一轮的计划与内容生成。
5. 越多内容被引用 → 智能增益越高、最优打法越清晰 → **越用越准**（这就是 PRD 的"数据资产飞轮"/护城河）。

**前端**：工作台 `Dashboard.tsx` 顶部「AI Agent 运作中枢」可视化这 7 个 Agent 闭环 + 学习面板(智能增益/最优打法/下一轮短板)。学习信号经 `PipelineContext.learningFor()` 提供，`generateFromPlan()` 生成时已采纳。

**后端要接**：优化 Agent = 定时作业，按 `mention_records`(被引结果) 聚合 (platform×contentType×intent) 的命中率，更新一张 `playbook` 权重表；计划/内容生成读它做加权（轻量"规则引擎+A/B"，不必硬上强化学习，符合 PRD §8.6.3）。
