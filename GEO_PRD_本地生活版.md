## 第 0 部分：执行摘要（TL;DR）

**一句话定位**：用 AI 给本地生活商家做"AI 搜索时代的 SEO"，让用户在 DeepSeek / 豆包 / Kimi / 元宝 / ChatGPT 里问"推荐一家 XXX"时，品牌被优先提及、被准确描述、被引用为答案。

**核心判断（vibe coding 可行性）**：
- **MVP 完全可做**：用 Cursor + Claude + Supabase + Next.js + 主流大模型 API，1 人 6–10 周可上线最小闭环。
- **不能完整对标迈富时**：他们的护城河是 "10 年营销服务+股东背景+全平台账号资源+B 端销售铁军"，不是技术。技术层面（诊断、Prompt 库、知识库 RAG、内容生成、看板）你完全能做；难做的是**外部平台内容分发的合规化运营**和**B 端销售网络**。
- **垂直行业是你最大优势**：迈富时是横向通用平台（什么行业都做），你聚焦本地生活三个细分行业，**模板更深、效果更实、单店付费意愿更明确**。这是典型的"在大象腿边长草"的好生意。

**两阶段商业路径**：

| 阶段 | 周期 | 核心动作 | 收入模型 | 关键产出 |

## 1. 项目背景与市场机会

### 1.1 搜索流量的范式迁移

2024 年起，全球用户的"决策性搜索"正在从传统搜索引擎迁移到生成式 AI：
- 国内：DeepSeek、豆包、Kimi、元宝、文心一言、通义千问、智谱清言
- 国外：ChatGPT、Gemini、Claude、Perplexity、Grok
- 入口型：抖音 AI 搜、小红书 AI 搜、百度 AI 智能问答、微信 AI 搜

用户问的不再是"上海火锅"，而是"上海徐汇区适合 4 个人聚餐、人均 200 以内、有包间的火锅推荐"——AI 直接给出一段带品牌名的推荐答案，**点击量大幅集中到被 AI 提及的前 3–5 个品牌**。

### 1.2 本地生活商家的真实痛点

我们调研虚拟样本（3 城 × 3 行业 × 各 5 家店）后归纳：

| 痛点 | 餐饮 | 按摩 SPA | 酒旅 |
|---|---|---|---|
| AI 答案中"隐身" | ★★★★★ | ★★★★★ | ★★★★ |
| 信息失实/过时 | ★★★★ | ★★★ | ★★★★★ |
| 被同行（甚至已倒闭店）压制 | ★★★★ | ★★★★★ | ★★★ |
| AI 错配标签（如把高端店写成快餐） | ★★★ | ★★★★ | ★★★★ |
| 不知道怎么"优化" | ★★★★★ | ★★★★★ | ★★★★★ |
| 没有持续维护的内容资产 | ★★★★★ | ★★★★★ | ★★★★ |

### 1.3 市场规模粗算

- 中国本地生活商家：餐饮 ~800 万家、按摩/美容/SPA ~150 万家、酒店民宿 ~80 万家。
- 假设可触达 1%（即 ~10 万家中型以上商家），ARPU 12,000 元/年。
- TAM = 10 万 × 12,000 = **12 亿/年**（仅三个行业）。
- 即便你只拿 1% 市场份额，也是 1,200 万/年的生意。

### 1.4 为什么是现在

- AI 搜索月活已破 5 亿（国内），且仍在高速增长。
- 商家普遍意识到"百度排名"的钱白花了，但还没找到 AI 时代的新解法——**认知教育窗口期已到来**。
- 大模型 API 价格在 2025–2026 年降到了几乎可忽略的水平（Claude Haiku / DeepSeek 输入 < 1 元/百万 token），允许 vibe coder 跑大量诊断任务而不亏钱。

```

   ↑ 意图建模         ↑ 语料工程          ↑ 场景强化
```

| 模块 | 迈富时叫法 | 本质 |

| 02 | 语义工程逆向解析与 Prompt | 反推 AI 优质回答逻辑，建高权重 Prompt 库 |

| 05 | 数据总览与优化引擎 | 多平台监控 + 强化学习闭环优化 |

## 3. Vibe Coding 可行性评估

### 3.1 能做 vs 难做（一张表说清）

| 能力模块 | Vibe Coding 难度 | 推荐工具/技术栈 | 备注 |
|---|---|---|---|
| 前端 UI（看板、表单、报告） | ⭐ 极易 | Next.js + Tailwind + shadcn/ui + Cursor | 一周搞定 MVP |
| 用户/客户/订单 CRUD | ⭐ 极易 | Supabase（DB+Auth+Storage 一体）| 直接用 |
| AI 诊断引擎（多模型跑测） | ⭐⭐ 容易 | 调用大模型 API + 任务队列（Trigger.dev / Inngest）| 见 §3.2 |
| Prompt 库管理 | ⭐⭐ 容易 | Postgres + 简单版本控制 | |
| 知识库 RAG | ⭐⭐⭐ 中等 | Supabase pgvector / Pinecone + LangChain | 复杂度可控 |
| 内容生成 | ⭐⭐ 容易 | Claude/GPT/DeepSeek API | 套 prompt |
| 数据看板 | ⭐⭐ 容易 | Recharts / Tremor + Supabase | |
| 报告 PDF 导出 | ⭐⭐ 容易 | Puppeteer / React-PDF | |
| **多 AI 平台爬取**（无 API 的：豆包/元宝/Kimi 等） | ⭐⭐⭐⭐ 较难 | Playwright + 代理池 + 验证码识别 | **关键难点 1** |
| **外部平台内容分发**（小红书/抖音/大众点评） | ⭐⭐⭐⭐⭐ 极难 | 需账号矩阵+人工辅助+合规风险 | **关键难点 2** |
| 多租户 SaaS 化 | ⭐⭐⭐ 中等 | Row-Level Security + 计费（Stripe/微信支付）| 阶段二再做 |
| 强化学习闭环 | ⭐⭐⭐⭐ 难 | 简化为"规则引擎+A/B"即可 | 不要硬做 RL |

### 3.2 关于"多 AI 平台监测"的工程现实

迈富时宣称"主流 AI 平台"覆盖，但实际：
- 有 API 的（容易）：OpenAI、Claude、Gemini、DeepSeek、月之暗面 Kimi（开放平台版）、通义、智谱
- 半开放（中等）：文心一言（部分 API）、豆包（火山方舟）、混元
- 无稳定 API、需爬虫（难）：豆包 C 端 App、Kimi C 端 Web/App、元宝 C 端、AI 搜索入口（抖音搜、小红书搜）

**Vibe Coding 实战策略**：
- 优先用 API（一周可接 10+ 个模型）。
- 难爬的 C 端用 Playwright + 代理池模拟，**接受 ~85% 成功率**，定时重试。
- 对客户报告说"覆盖主流 AI 平台 X 个"即可，不要承诺"100% 实时全覆盖"。

### 3.3 关于"内容分发"的真相

这是迈富时也没真正解决的事，也是行业最大灰色地带：
- 小红书、抖音、大众点评对营销内容打压严格。
- 不能纯自动化发布，必须**矩阵账号 + 人工辅助 + 内容差异化**。
- 商业落地形态：**给客户出一个内容包（图文/视频脚本/达人合作话术），让客户/达人发布，而不是自动发**。

### 3.4 最终可行性结论

**完全可行，但要正确定位**：
- ✅ 能做：监测、诊断、知识库、Prompt 库、内容生成、报告、看板、SaaS。
- ⚠️ 半做：内容分发（人机协作）、多 AI 平台爬取（部分模拟）。
- ❌ 不做：自动化养号、刷量、纯黑帽（合规风险）。

**MVP 资源配置建议**：1 个产品/创始人 + 1 个 vibe coder（你自己也行）+ 1 个内容运营/客户成功。**初期 6–10 周上线**。

## 5. 产品定位与愿景

### 5.1 一句话定位

> **"本地生活的 AI 搜索增长引擎——让你的店，被 AI 优先推荐。"**

### 5.2 三句话产品逻辑

1. **诊断**：告诉你"现在 AI 怎么看你的店"——在哪些场景被推荐、被怎么描述、和谁竞争。
2. **构建**：为你建立"AI 友好"的内容与信源资产——知识库、Prompt 库、E-E-A-T 内容包。
3. **优化**：持续投喂、持续监测、持续优化——直到 AI 在关键场景把你排到前 3。

### 5.3 愿景（3 年）

成为中国本地生活商家 AI 时代的"营销基础设施"，签约 5,000 家店，年收入过亿，沉淀本地生活领域最完整的"AI 推荐机制"数据资产。

### 5.4 价值主张（对客户讲清楚的话）

| 客户怀疑 | 我们的回答 |

## 6. 商业模式

### 6.1 阶段一：自营代运营（0–9 个月）

**产品形态**：内部工具 + 服务包

**定价方案**：

| 套餐 | 客单价（季度） | 服务内容 | 适用客户 |
|---|---|---|---|
| 启动包 | 6,800 元 | 1 次诊断报告 + 知识库构建 + 1 个月监测 | 单店/试水 |
| 标准包 | 19,800 元 | 完整诊断 + 知识库 + Prompt 库 + 月度内容包 ×3 + 季度优化报告 | 单店深度 |
| 旗舰包 | 39,800 元 | 上面所有 + 周度监测 + 达人合作话术 + 内容投喂代发 + 复盘会 | 高客单店/小连锁 |
| 集团/连锁定制 | 8–30 万/年 | 多店管理 + 专属客户成功 | 5+ 店连锁 |

**毛利结构**（标准包 19,800 测算）：
- 大模型 API 成本：~200 元
- 内容生产人工：~3,000 元
- 客户成功人工：~2,000 元
- 工具/账号/代理：~500 元
- 毛利 ≈ 14,000 元（70%+）

### 6.2 阶段二：SaaS 化（9–24 个月）

**产品形态**：多租户 SaaS + 行业模板市场 + 服务包加购

**定价方案**：

| 套餐 | 月费 | 功能 |
|---|---|---|
| 单店自助 | 599 元 | 1 店、5 AI 平台监测、月 1 次诊断、内容建议 |
| 单店专业 | 1,499 元 | 1 店、10 AI 平台、周诊断、知识库、Prompt 库、内容生成 ×8/月 |
| 多店连锁 | 2,999 元起（按店阶梯） | 3 店起、集团看板、API、定制行业模板 |
| 白标渠道 | 询价 | 代运营机构白标接入 |
| 服务包加购 | 3,000–15,000 元/月 | 内容代运营、达人对接、深度报告 |

**预估收入模型（24 个月）**：
- 自营 30 客户 × 80,000 元/年 = 240 万
- SaaS 200 客户 × 18,000 元/年 = 360 万
- 服务包加购 = 200 万
- **24 个月总收入 ~800 万**

### 7.2 设计原则

1. **行业模板优先**：底层通用，顶层用模板差异化。新增行业 = 加一套模板。
2. **结果可视化**：客户每月能拿到一份"看得懂"的报告，里面有进步数字。
3. **人机协作**：能 AI 做的让 AI 做，必须人工的不勉强自动化（避免合规风险）。
4. **资产沉淀**：所有诊断、Prompt、知识、内容都进库，越用越值钱（也是续费理由）。
5. **从服务出 SaaS**：所有 SaaS 功能都是先在自营中验证过的"高频痛点"。

## 8. 功能模块详细设计

### 8.1 模块 ①：AI 诊断引擎（Brand AI Diagnosis）

#### 8.1.1 价值
一键告诉客户"AI 怎么看你的店"——是否被提及、被怎么描述、被谁压制。

#### 8.1.2 核心功能

**8.1.2.1 关键词/意图矩阵生成**
- 输入：店铺基本信息（名称、城市、品类、特色）
- 输出：50–200 个目标 AI 搜索意图（按行业模板生成）
- 实现：基于行业模板 + Claude 生成长尾意图

**8.1.2.2 多 AI 平台并发跑测**
- 覆盖平台（MVP）：DeepSeek、豆包、Kimi、元宝、ChatGPT、Gemini、文心一言、通义千问、智谱清言、Claude（10 个）
- 每个意图在每个平台跑 3 次（消除随机性），取并集
- 抓取：是否提及品牌、提及位置（排名）、上下文描述、引用信源、竞品提及

**8.1.2.3 评分模型（T-LBR™ 指数）**

我们的对标模型，建议命名 **T-LBR™（Local Brand Recommendation Index）**：

```
LBR Score = 0.35 × 提及率
          + 0.25 × 推荐排名得分（排第 1=10, 第 2=8, 第 3=6, 第 4-5=4, 后=1）
          + 0.20 × 描述准确度（NLP 比对官方信息）
          + 0.10 × 信源权威度（被引用平台权重）
          + 0.10 × 情感倾向（正向/中性/负向）

按 4 大维度展示：曝光、定位、信任、转化倾向
```

**8.1.2.4 竞品对照**
- 自动识别 AI 答案中被提及的同行
- 生成"你 vs 前 5 名竞品"矩阵
- 标红"该被推却没被推"的场景

**8.1.2.5 诊断报告自动生成**
- PDF 报告（30–50 页）+ Web 在线版
- 包含：现状评分、差距分析、行业基准、优化建议、优先级
- 自动套行业模板（餐/SPA/旅各一套）

#### 8.1.3 用户故事
> 作为火锅店老板老王，我希望上传我店的信息后 24 小时内拿到一份报告，告诉我"我现在在 10 个 AI 平台上的露出情况"，并且告诉我"哪些场景我没被推、为什么、怎么优化"。

#### 8.1.4 关键技术点

```python
# 伪代码示意
async def diagnose_brand(shop_id):
    shop = get_shop(shop_id)
    intents = generate_intents(shop, template=shop.industry)   # 100 个意图
    platforms = ["deepseek", "doubao", "kimi", "yuanbao", ...]
    
    tasks = []
    for intent in intents:
        for platform in platforms:
            tasks.append(query_platform(platform, intent, runs=3))
    
    results = await asyncio.gather(*tasks)   # 并发跑
    parsed = [parse_response(r, shop.name, competitors=shop.competitors) for r in results]
    
    score = calculate_lbr(parsed)
    report = generate_report(shop, parsed, score)
    return report
```

API 调用预算（单次诊断）：100 意图 × 10 平台 × 3 次 ≈ 3,000 次调用，约 50–200 元成本。

### 8.2 模块 ②：Prompt 工程库（Prompt Engineering Hub）

#### 8.2.1 价值
研究"AI 优质回答的逻辑"，把它结构化为可复用的 Prompt 模板，作为内容生产、知识库构建的底层资产。

#### 8.2.2 核心功能

- **逆向解析**：对 AI 答案做反推——为什么 AI 推荐的是这家？答案里出现了哪些"高权重表达"？
- **Prompt 模板库**：按"行业 × 场景 × 目的"分类，初始库存 500+ 模板。
- **A/B 测试**：同一目标，两个 prompt，输出对比，留优汰劣。
- **Prompt 版本管理**：Git-like，每次修改有 diff，可回滚。
- **客户专属 Prompt**：每个客户基于通用模板派生专属版本（含店铺关键词、卖点）。

#### 8.2.3 Prompt 模板示例（餐饮）

```
# 模板 ID: FB-INTRO-01
# 用途：生成被 AI 引用的"店铺一段话介绍"
# 适用场景：百科、问答、社媒底纹内容

请用 200 字内介绍一家餐厅，要求：
1. 第 1 句直击品类与差异化（如"主打 XX 派系"、"广州第一家 XX"）
2. 第 2 句给出 1 个最具辨识度的招牌菜及其特征
3. 第 3 句给出权威背书（媒体提名/奖项/老字号年份/主厨经历）
4. 第 4 句给出场景适配（适合什么人、什么场合）
5. 第 5 句结束于体验关键词（人均、营业时间、地址）
不要营销话术（如"绝佳/超棒/震撼"），用事实陈述。
信息源：{shop_info}
```

#### 8.2.4 数据模型片段（Postgres）

```sql
CREATE TABLE prompt_templates (
  id UUID PRIMARY KEY,
  code TEXT UNIQUE,              -- FB-INTRO-01
  industry TEXT,                  -- restaurant | spa | hotel
  scenario TEXT,                  -- intro | review | menu | etc.
  purpose TEXT,                   -- 用途说明
  body TEXT,                      -- prompt 模板正文
  variables JSONB,                -- 可填变量
  version INT,
  parent_id UUID,                 -- 派生关系
  metrics JSONB,                  -- 历史表现（生成质量分、被引率）
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ
);
```

### 8.3 模块 ③：行业知识库（Industry Knowledge Base，RAG）

#### 8.3.1 价值
把客户店铺的所有非结构化资料（菜单 PDF、官网、点评摘录、视频文案、媒体报道）转化为**AI 可理解、可推理、可被引用的结构化知识**。

#### 8.3.2 知识图谱结构（餐饮示例）

```
店铺（Shop）
├── 基本信息（地址、电话、营业时间、人均、停车、地铁距离）
├── 菜系/派系（粤菜·顺德 / 川菜·正宗川味 / 日料·寿司）
├── 招牌菜 ×N（每道菜：菜名、价格、做法、原料、口味、推荐场景）
├── 主厨/创始人（背景、奖项、师承）
├── 历史/故事（创立年份、品牌故事、扩张历史）
├── 场景适配（聚餐、约会、商务、生日、独食…）
├── 设施特色（包间数、最大容人、无烟、宠物友好、儿童椅、停车…）
├── 媒体背书（米其林、黑珍珠、媒体报道、奖项）
├── 用户口碑摘要（来自点评的高频词、好评点、差评点）
└── 周边环境（附近地标、交通、停车）
```

SPA、酒旅各有独立的图谱模板（详见 §9）。

#### 8.3.3 核心功能

- **资料上传**：PDF、Word、图片、URL、文本，全部入库
- **AI 抽取**：用 Claude/GPT 把非结构化资料抽成图谱节点
- **人工审核**：客户/运营审核确认，准确率 95%+ 才发布
- **向量化与检索**：pgvector，支持语义检索
- **被 AI 引用追踪**：哪些字段被 AI 答案引用了？引用率多高？
- **资产估值**：知识库越完整，T-LBR™ 越高，给客户看"资产增值曲线"

#### 8.3.4 技术架构

```
原始资料 → AI 抽取（Claude）→ 结构化字段 → 人工审核 → 
  ↓
  ├──→ Postgres（结构化存储）
  ├──→ pgvector（语义检索）
  └──→ 知识卡片（前端展示 + 投喂内容生产）
```

- ❌ **不刷量、不买虚假评论**

### 9.2 按摩 SPA 行业

#### 9.2.1 行业特征（最难也最有钱赚）
- **信任决策极强**：用户最大顾虑是"会不会踩雷/会不会是黑店"
- **隐私敏感**：很多用户不敢去陌生店
- **AI 容易"污名化"**：行业关键词容易触发模型的安全过滤，被 AI 弱化推荐
- **客单价高、复购强**：一旦信任建立，LTV 极高

#### 9.2.2 核心挑战与对策

| 挑战 | 对策 |
|---|---|
| AI 把按摩行业关键词降权或屏蔽 | 用"养生 / 推拿 / 中医调理 / 身心放松"等替代词构建知识图谱 |
| 用户"是否正规"的顾虑 | 强化资质、合规、卫生标准、技师培训等 E-E-A-T 信号 |
| 行业内灰色玩家多，竞争同质 | 用"健康功效 + 真实疗程 + 主理人故事"做品牌差异化 |

#### 9.2.3 知识图谱关键字段

```
Shop (SPA / Wellness)
├── 基础信息
├── 流派 (中医推拿 / 泰式 / 精油 / 足道 / 美容 SPA / 物理治疗)
├── 招牌项目 [Service] (name, duration, price, target, technique, contraindications)
├── 技师 [Therapist] (name, certificate, years_of_experience, specialty) ★关键★
├── 资质证照 (business_license, hygiene_certificate, traditional_chinese_medicine_cert)
├── 卫生标准 (one_use_disposable, sanitation_process, towel_policy)
├── 环境 (single_room, couple_room, female_only, atmosphere)
├── 适用人群 (couple, pregnancy, elderly, athlete, office_worker)
├── 健康功效 (cervical, lumbar, sleep, immunity, recovery)
├── 媒体/权威背书 (中医协会、康复医学、媒体报道)
└── 安全合规承诺
```

#### 9.2.4 核心 KPI
- "正规 / 不踩雷 / 推荐"类意图 Top 3 命中率（核心）
- 信任类关键词覆盖率（资质、卫生、技师经验）
- 招牌项目被 AI 描述准确度

#### 9.2.5 内容投喂优先级

### 10.2 关键技术决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 单体 vs 微服务 | **单体起步** | vibe coding 1 人维护，单体最快 |
| 多租户隔离 | **Row-Level Security**（Supabase 原生）| 比起多 schema/多库省 90% 工作量 |
| AI 模型选择 | **多模型组合**（DeepSeek 跑量+Claude 提质+GPT 兜底）| 成本可控 |
| RAG 引擎 | **不用 LlamaIndex/LlamaCloud**，pgvector + 简单代码 | 控制成本+可控性 |
| 爬虫稳定性 | **接受 80–90% 成功率，重试+人工兜底** | 完美没必要 |
| 数据合规 | **国内客户数据存阿里云国内节点**，模型走合规 API（DeepSeek、豆包国内版）| 必须 |

### 12.1 自营阶段：新客户全流程


### 12.2 SaaS 阶段：自助用户流程

```
01 注册 → 02 选套餐付费 → 03 引导填写店铺资料 → 04 一键诊断 →
05 看在线报告 → 06 系统每周生成建议 → 07 内容工厂生成 → 
08 客户/达人发布 → 09 数据自动追踪 → 10 月度账单
```

### 12.3 诊断流程时序（核心技术流程）

```
1. 客户提交店铺信息
2. 系统选行业模板，生成 100 意图
3. 任务队列拆分：100 × 10 平台 × 3 次 = 3000 任务
4. 并发执行（API + Playwright 混合）
5. 结果入 intent_queries 表
6. NLP 解析：是否提及、排名、描述、信源、竞品
7. 计算 LBR
8. 生成报告（Web + PDF）
9. 推送给客户成功 + 客户
```

| 坑 | 后果 | 怎么避 |

## 16. 商业化设计

### 16.2 续费机制

- **季度合同**：付 1 季送 1 个月（鼓励 4 个月起步）

4. **达人合作引流**：MCN 网络反向带客
5. **门店地推**（高客单密度区，如餐饮街、酒店带）

### 16.4 渠道生态

- **本地代运营公司**：白标 + 分成（20–40%）

## 18. 风险与缓解

### 18.1 主要风险矩阵

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|

## 20. 附录

### 20.1 名词与缩写

- **GEO**：Generative Engine Optimization（生成引擎优化）
- **T-LBR™**：Local Brand Recommendation Index（本地品牌推荐指数，本项目自有评估模型，对标 T-GEO™）
- **E-E-A-T**：Experience / Expertise / Authoritativeness / Trustworthiness（Google 内容评估标准）
- **TopRec@3**：在目标意图列表中，被 AI 推荐为前 3 名的比例

### 20.2 关键决策清单（创始人需在 1 周内拍板）

- [ ] 公司主体 / 品牌名 / 域名
- [ ] 是否海外节点（覆盖海外华人本地生活）
- [ ] 三行业首批合作客户名单（≥9 家意向）
- [ ] Vibe Coder 是创始人本人还是外聘
- [ ] 第一个种子客户/灯塔客户（强烈建议挑一个有"故事"的店）
- [ ] 商标 + T-LBR™ 注册（早申请）

### 20.3 0–30 天行动清单（创始人可以马上做的事）

| 天 | 动作 |
|---|---|
| Day 0–3 | 写一页 BP，找 3 个候选客户聊需求（验证付费意愿）|
| Day 4–7 | 用 Cursor 跑通"AI 诊断"原型（哪怕只针对你自己的店）|
| Day 8–14 | 完成餐饮模板 v0.1（50 意图 + 5 个 Prompt）|
| Day 15–21 | 接 1 个意向客户做免费诊断，输出第一份报告 |
| Day 22–30 | 签约第 1 个付费客户（标准包 19,800）|

### 20.4 关于"是否要做"的最后建议

**做。** 理由：

1. **大势**：AI 搜索的范式迁移是确定的（不是赌方向，是赌速度）。
2. **真实需求**：本地生活商家对"被 AI 推荐"的焦虑是真实的，迈富时已经验证了 B 端付费意愿。
3. **你的优势**：垂直行业 + 轻团队 + Vibe Coding，能跑得比大厂快 5 倍。
4. **可控风险**：MVP 投入 ≤30 万、6 个月可验证。
5. **退出路径**：即便不成 SaaS，做成"本地生活 GEO 代运营"也是个利润 30–50 万/年的小生意。

**不做**的唯一理由：你 6 个月内拉不来 5 个付费意向客户。如果连这个都做不到，不是行业不行，是销售不行——SaaS 也救不了。
